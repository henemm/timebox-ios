# RW 3.2 — Focus Sprint ("Los"-Button) Implementation Spec

## Approval

- [x] Approved

## Purpose

Enables a one-tap Focus Sprint directly from Backlog and Next-Up views. A single "Los" tap on any task creates a new Focus Block (start = now, duration = task's `estimatedDuration` or 60 min default), assigns the task to it, switches the app to the Focus tab, and starts the Live Activity via `FocusLiveView`. Conflicts (already-active block) are shown as an alert.

This spec covers the concrete implementation across 6 production files and 2 test files. It is derived from the feature spec at `docs/specs/rework/3.2-focus-sprint.md`.

### Step 1 — Service Layer (~65 LoC, 1 file)

#### `Sources/Services/FocusBlockActionService.swift`

Add a new `FocusSprintResult` enum and a `startImmediate()` static method to the existing `FocusBlockActionService` enum. Insert after `followUpTask()`.

**New `FocusSprintResult` enum:**

```swift
/// Result of a Focus Sprint start attempt.
enum FocusSprintResult: Sendable, Equatable {
    /// Sprint started successfully. blockID is the new EKEvent ID.
    case started(blockID: String)
    /// Another block is currently active — user must stop it first.
    case blockedByActiveBlock(blockTitle: String)
}
```

**New `startImmediate()` method:**

```swift
/// Start a Focus Sprint immediately for a given task.
/// Creates a new Focus Block (start = now, end = now + durationMinutes),
/// assigns the task, and persists assignedFocusBlockID on LocalTask.
///
/// - Parameters:
///   - taskID:          ID of the LocalTask to sprint on.
///   - eventKitRepo:    Repository for EKEvent operations (injected for testability).
///   - modelContext:    SwiftData context (injected for testability).
///   - durationMinutes: Duration override. If nil, uses task's estimatedDuration, else 60.
/// - Returns: `.started(blockID:)` or `.blockedByActiveBlock(blockTitle:)`.
@MainActor
static func startImmediate(
    taskID: String,
    eventKitRepo: any EventKitRepositoryProtocol,
    modelContext: ModelContext,
    durationMinutes: Int? = nil
) throws -> FocusSprintResult {

    // 1. Fetch today's blocks and check for an already-active one
    let todayBlocks = (try? eventKitRepo.fetchFocusBlocks(for: Date())) ?? []
    if let activeBlock = todayBlocks.first(where: { $0.isActive }) {
        return .blockedByActiveBlock(blockTitle: activeBlock.title)
    }

    // 2. Resolve duration: parameter → task.estimatedDuration → 60 min default
    let descriptor = FetchDescriptor<LocalTask>()
    let allTasks = (try? modelContext.fetch(descriptor)) ?? []
    guard let task = allTasks.first(where: { $0.id == taskID }) else {
        // Task not found — treat as no-op (safe fallback)
        return .blockedByActiveBlock(blockTitle: "")
    }

    let resolvedDuration = durationMinutes ?? task.estimatedDuration ?? 60

    // 3. Create the block (start = now, end = now + duration)
    let startDate = Date()
    let endDate = startDate.addingTimeInterval(TimeInterval(resolvedDuration * 60))
    let blockID = try eventKitRepo.createFocusBlock(startDate: startDate, endDate: endDate)

    // 4. Assign the task to the block
    try eventKitRepo.updateFocusBlock(
        eventID: blockID,
        taskIDs: [taskID],
        completedTaskIDs: [],
        taskTimes: [:]
    )

    // 5. Persist assignedFocusBlockID on LocalTask
    task.assignedFocusBlockID = blockID
    task.isNextUp = false   // Remove from Next Up — it's now in a sprint
    try? modelContext.save()

    return .started(blockID: blockID)
}
```

**Note:** `fetchFocusBlocks(for:)` already exists on `EventKitRepositoryProtocol`. The method signature is `func fetchFocusBlocks(for date: Date) throws -> [FocusBlock]`. Verify this signature before implementation — if the method is async, wrap the call in `try await` and change `startImmediate` to `async throws`.

**New helper method** (add alongside `updateNextUp` etc.):

```swift
private func startFocusSprint(for item: PlanItem) {
    do {
        let result = try FocusBlockActionService.startImmediate(
            taskID: item.id,
            eventKitRepo: eventKitRepo,
            modelContext: modelContext
        )
        switch result {
        case .started:
            focusSprintFeedback = true
            NotificationCenter.default.post(name: .focusSprintStarted, object: nil)
        case .blockedByActiveBlock(let title):
            focusSprintConflictTitle = title
        }
    } catch {
        errorMessage = "Focus Sprint konnte nicht gestartet werden: \(error.localizedDescription)"
    }
}
```

## Acceptance Criteria

- [ ] Tapping "Los" on a Backlog task with no active block creates a Focus Block and switches to the Focus tab
- [ ] The Focus tab shows the new block in `FocusLiveView` immediately (block is active, `isActive == true`)
- [ ] Duration = task's `estimatedDuration` when set; falls back to 60 min when `nil`
- [ ] `LocalTask.assignedFocusBlockID` is set to the new blockID after the tap
- [ ] `LocalTask.isNextUp` is set to `false` after the tap
- [ ] If a block is already active, a conflict alert appears with the active block's title — no new block is created
- [ ] Tapping "Los" on a Next-Up task triggers the same flow
- [ ] On macOS, "Focus Sprint starten" appears in the BacklogRow context menu
- [ ] The "Los" button is only visible when the callback is wired (no visual regression for callers that don't use it)
- [ ] No second parallel Focus Block is ever created

## Expected Behavior

- **Input:** User taps "Los" on a `PlanItem` in `BacklogRow` or `NextUpRow`
- **Output:**
  - New EKEvent (Focus Block) created via `EventKitRepository.createFocusBlock`
  - Task assigned via `updateFocusBlock(eventID:taskIDs:completedTaskIDs:taskTimes:)`
  - `LocalTask.assignedFocusBlockID` persisted in SwiftData
  - `Notification.Name.focusSprintStarted` posted → `FocusBloxApp` switches to `.focus` tab
  - `FocusLiveView` appears; its existing `loadData()` on `.onAppear` picks up the new active block
- **Side effects:**
  - `LocalTask.isNextUp` cleared
  - Haptic feedback (`.impact(weight: .medium)`) on success
  - Conflict alert shown if another block is active
  - Live Activity + Dynamic Island activated by `FocusLiveView` (existing behavior, no new code needed)

## Known Limitations

- **`fetchFocusBlocks(for:)` signature:** Must verify whether this method is `throws` or `async throws` on `EventKitRepositoryProtocol`. If async, `startImmediate` becomes `async` and all callers must use `Task { try await ... }`.
- **Duration picker:** No inline picker before sprint start. The sprint always uses `estimatedDuration ?? 60`. Follow-up: FEATURE_023.
- **macOS tab switch:** `FocusBloxApp.swift` is iOS-only. On macOS, the notification is not posted from `FocusBloxApp`. The macOS root view (`MacContentView` or equivalent) must add a matching `.onReceive` observer to switch the sidebar selection to `.focus`. This is out of scope for RW 3.2.
- **`NotificationCenter` + `@MainActor`:** `startFocusSprint` in `BacklogView` is called from a SwiftUI button action (already on main actor). `NotificationCenter.default.post` is synchronous and runs on the caller's thread — no additional dispatch needed.
- **EKEvent title:** The created block uses `FocusBlock.generateTitle(for: startDate)` (e.g., "FocusBlox 14:32"). No task-title is embedded in the event title in this scope.

## Implementation Deviations

The following deviations from the original spec were made during implementation:

### 1. BacklogView uses BacklogRow for Next-Up tasks (not NextUpSection/NextUpRow)
BacklogView renders Next-Up tasks inside its own `Section` using `BacklogRow` — not via the standalone `NextUpSection` component. Consequently, Next-Up rows expose `focusSprintButton_` identifiers (same as regular Backlog rows), NOT `nextUpSprintButton_` identifiers. The `NextUpSection.swift` file exists with `nextUpSprintButton_` but is not used by BacklogView.

### 2. `.buttonStyle(.borderless)` instead of `.buttonStyle(.plain)`
The sprint button in `BacklogRow` uses `.buttonStyle(.borderless)` to ensure button taps are correctly routed in SwiftUI Lists with multiple interactive elements per row. `.buttonStyle(.plain)` caused tap-handling conflicts.

### 3. `.contentShape(Rectangle())` removed from BacklogRow HStack
The outer HStack's `.contentShape(Rectangle())` was removed because it intercepted taps intended for the sprint button. Without it, button hit-testing works correctly.

### 4. `.safeAreaInset(edge: .bottom)` added to BacklogView list
A 120pt bottom safe area inset was added to the list to ensure the last row (and its sprint button) can be scrolled into the hittable area above the tab bar.

### 5. Section `.accessibilityIdentifier` moved to header HStack
Applying `.accessibilityIdentifier("nextUpSection")` directly on `Section` caused ALL child element identifiers to be overridden. Moved to the header's `HStack` to preserve child identifiers.

### 6. `--no-active-block` launch argument instead of `--inject-sprint-tasks`
The spec proposed `--inject-sprint-tasks`. Implementation uses `--no-active-block` (removes the active mock block) because existing `-UITesting` mock data already includes all needed tasks. This approach is simpler — it controls the presence/absence of the active block rather than injecting additional data.

### 7. `focusLiveView_root` test omitted
The `testTappingFocusSprintButton_showsFocusLiveView` test was omitted because `MockEventKitRepository.createFocusBlock` does not add the newly created block to `mockFocusBlocks`. The tab-switch test verifies the full notification flow instead.
