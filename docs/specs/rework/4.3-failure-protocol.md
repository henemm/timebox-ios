# Story 4.3: Failure Protocol

**Epic:** 4 — Tagesabschluss & Reflexion
**Priorität:** Mittel | **Abhängigkeiten:** Story 4.1

## Ziel

Optionales Quick-Select: Warum wurde ein Task nicht erledigt? Daten fließen in BehavioralProfileService und NextUpSuggestionService.

## Akzeptanzkriterien

- [ ] Im Abend-Modus der Tagesansicht: Unerledigte Tasks mit Quick-Select
- [ ] 6 vordefinierte Gründe: `noTime`, `tooTired`, `blocked`, `notRelevant`, `forgotAboutIt`, `other`
- [ ] Komplett optional (wegwischbar)
- [ ] Gründe werden historisiert (Datum + Grund)
- [ ] BehavioralProfileService konsumiert Failure-Daten für bessere Vorschläge:
  - `tooTired` → Task nicht morgens vorschlagen wenn Profil zeigt: abends müde
  - `noTime` → Task an Tagen mit weniger Meetings bevorzugen
- [ ] Aggregierte Statistik im DayView: "Häufigste Gründe diese Woche" (optional)

## Technische Vorgaben

### Neues Model
```swift
@Model
class TaskFailureRecord {
    var taskID: String
    var date: Date
    var reason: FailureReason

    enum FailureReason: String, Codable {
        case noTime, tooTired, blocked, notRelevant, forgotAboutIt, other
    }
}
```

### Neue Dateien
| Datei | Beschreibung |
|---|---|
| `TaskFailureRecord.swift` | SwiftData Model |
| `FailureProtocolService.swift` | Speichert, aggregiert, liefert Insights |
| `FailureQuickSelectView.swift` | 6 Buttons, dismissbar |

### Betroffene Dateien
| Datei | Änderung |
|---|---|
| `EveningReflectionSection.swift` | + FailureQuickSelectView pro unerledigtem Task |
| `BehavioralProfileService.swift` | + Failure-Daten als Input (wenn verfügbar) |
| `NextUpSuggestionService.swift` | + Failure-basierte Filter |

### Constraints
- FailureRecords: Append-only, nie löschen
- CloudKit-Sync
- Index auf `date` und `reason` für Performance
- UI: Neutrale Sprache ("Woran lag's?" nicht "Warum hast du versagt?")
