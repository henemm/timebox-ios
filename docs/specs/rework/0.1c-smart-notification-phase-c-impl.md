# SmartNotificationEngine Phase C — DueDate Migration

## Approval

- [ ] Approved

## Purpose

Replace all 14 remaining direct `NotificationService.scheduleDueDateNotifications` / `cancelDueDateNotifications` calls across Views and Delegates with `SmartNotificationEngine.reconcile()`. After Phase C, no View or Delegate calls NotificationService directly — only the Engine does.

## Source

- **File:** Multiple (see Implementation Details)
- **Identifier:** `SmartNotificationEngine.reconcile(reason:context:eventKitRepo:)` and `reconcile(reason:container:eventKitRepo:)`

## Dependencies

| Entity | Type | Reconcile Overload | Purpose |
|--------|------|-------------------|---------|
| SmartNotificationEngine (Phase A+B) | service | — | reconcile() overloads already implemented |
| NotificationService | service | — | build*Request methods used internally by Engine only |
| EventKitRepositoryProtocol | protocol | both | Required for reconcile — fetches FocusBlocks for budget |
| AppSettings | model | — | notificationProfile drives budget decisions inside Engine |
| ModelContainer | swiftdata | `reconcile(reason:container:eventKitRepo:)` | Used by NotificationActionDelegate (NSObject, kein @Environment) |
| ModelContext | swiftdata | `reconcile(reason:context:eventKitRepo:)` | Used by Views via @Environment(\.modelContext) |

**Welche Komponente nutzt welchen Overload:**
- `NotificationActionDelegate` → `reconcile(reason:container:eventKitRepo:)` (hat ModelContainer via init)
- `BacklogView`, `TaskFormSheet`, `CreateTaskView` → `reconcile(reason:context:eventKitRepo:)` (haben ModelContext via @Environment)
- `FocusBloxMac/ContentView` → `reconcile(reason:context:eventKitRepo:)` (hat ModelContext via @Environment)

**Alle Aufrufer von `NotificationActionDelegate.init`:**
- `Sources/FocusBloxApp.swift` (iOS App Entry Point)
- `FocusBloxMac/FocusBloxMacApp.swift` (macOS App Entry Point)
- Keine Tests, keine Factories — nur diese zwei Stellen

## Implementation Details

### Files to Modify (7 total)

#### 1. `Sources/Services/NotificationActionDelegate.swift` (MODIFY — 5 calls)

Add `eventKitRepository: any EventKitRepositoryProtocol` as second init parameter. Replace all cancel/schedule pairs and standalone cancels with a single reconcile call after each data mutation:

```swift
// REMOVE (lines 63–64, 71–72, 84):
NotificationService.cancelDueDateNotifications(taskID: task.id)
NotificationService.scheduleDueDateNotifications(taskID: ..., title: ..., dueDate: ...)

// ADD after each mutation:
await SmartNotificationEngine.reconcile(
    reason: .taskChanged,
    container: container,
    eventKitRepo: eventKitRepository
)
```

Affected actions: `actionPostponeTomorrow`, `actionPostponeNextWeek`, `actionComplete`.

### Implementation Order

1a. `NotificationActionDelegate.swift` — extend init + replace 5 calls
1b. `FocusBloxApp.swift` + `FocusBloxMacApp.swift` — update Delegate init calls (abhaengig von 1a — muss gleichzeitig compilieren)
2a. `BacklogView.swift` — replace 5 calls (unabhaengig)
2b. `FocusBloxMac/ContentView.swift` — replace 2 calls (unabhaengig)
2c. `TaskFormSheet.swift` — add @Environment + replace 1 call (unabhaengig)
2d. `CreateTaskView.swift` — add @Environment + replace 1 call (unabhaengig)

**Hinweis:** reconcile() ist idempotent — ein Partial-Rollback (z.B. nur 4/7 Dateien migriert) ist sicher. Nicht-migrierte Views planen dann Einzelnotifications, die beim naechsten Reconcile ueberschrieben werden.

## Expected Behavior

- **Input:** Any task mutation (create, edit, delete, postpone, complete) in any affected View or Delegate
- **Output:** `reconcile()` recalculates all 64 notification slots atomically — no inconsistent individual cancel/schedule pairs remain
- **Side effects:**
  - No direct `NotificationService.schedule/cancelDueDateNotifications` calls remain in any View or Delegate
  - `NotificationActionDelegate` init signature gains `eventKitRepository: any EventKitRepositoryProtocol` parameter
  - `TaskFormSheet` and `CreateTaskView` gain `@Environment(\.eventKitRepository)`
  - Both app entry points updated to pass `eventKitRepository` to `NotificationActionDelegate`

## Known Limitations

- `NotificationService.schedule/cancel` methods remain available (not deprecated) — they are used internally by SmartNotificationEngine; only external call sites are removed
- `buildReviewRequests()` and `buildNudgeRequests()` remain empty stubs until Phase D (RW_0.1d)
- Settings UI Profil-Picker deferred to Phase D (RW_0.1d)
- `NotificationActionDelegate` is `NSObject` + `@unchecked Sendable` — cannot use `@Environment`; `EventKitRepositoryProtocol` is injected via init instead
