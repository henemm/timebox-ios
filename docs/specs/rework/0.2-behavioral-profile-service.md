# Story 0.2: BehavioralProfileService

**Epic:** 0 — Infrastruktur
**Priorität:** Hoch | **Abhängigkeiten:** Keine (nutzt bestehende Daten)

## Ziel

Deterministischer Service, der über Wochen ein persönliches Verhaltensprofil des Nutzers aufbaut. Kein LLM, keine AI — reine Statistik auf bestehenden Daten. Andere Services (NextUpSuggestionService, EmotionalNudgeService, SuccessStoryService) konsumieren das Profil.

## Warum kein LLM?

Das Apple On-Device-Modell (~3B Parameter) ist optimiert für Textverarbeitung (Extraktion, Klassifikation, kurze Generierung). Es kann NICHT über temporale Muster reasonen oder Verhaltensvorhersagen treffen. Die Intelligenz hinter den Vorschlägen muss deterministisch sein — das LLM wird nur genutzt, um Ergebnisse in natürliche Sprache zu übersetzen (→ Story 4.2).

## Datenquellen (existieren bereits)

| Datenfeld | Quelle | Was es verrät |
|---|---|---|
| `completedDate` + Uhrzeit | `LocalTask` | Wann erledigt der Nutzer welche Tasks? |
| `category` | `LocalTask` | Welche Kategorien zu welchen Tageszeiten? |
| `estimatedDuration` vs. tatsächliche Dauer | `LocalTask` + `FocusBlock` | Wie gut schätzt der Nutzer? |
| `rescheduleCount` | `LocalTask` | Welche Tasks werden chronisch verschoben? |
| `importance` | `LocalTask` | Korrelation Importance ↔ Verschiebungsrate? |
| Kalender-Events | `EventKitRepository` | Wie beeinflusst Meeting-Dichte die Produktivität? |
| `TaskFailureRecord.reason` | Story 4.3 | Warum scheitern Tasks? (erst verfügbar nach Epic 4) |

## Akzeptanzkriterien

- [ ] Service berechnet folgende Profile (jeweils Rolling Window: letzte 28 Tage):
  - **Tageszeit-Affinität pro Kategorie:** Zu welcher Tageszeit erledigt der Nutzer welche Kategorie am häufigsten? (Vormittag/Nachmittag/Abend × 5 Kategorien)
  - **Tägliche Kapazität:** Durchschnittliche Anzahl erledigter Tasks und Gesamtdauer pro Tag
  - **Kalender-Korrelation:** Durchschnittliche Task-Completion an Tagen mit 0-2 / 3-4 / 5+ Meetings
  - **Schätz-Genauigkeit:** Verhältnis geschätzte vs. tatsächliche Dauer (Faktor)
  - **Verschiebungs-Muster:** Tasks mit rescheduleCount >= 3 + deren gemeinsame Merkmale (Kategorie, Importance, Tageszeit)
- [ ] Profil wird einmal täglich neu berechnet (beim ersten App-Start des Tages) und gecacht
- [ ] Profil ist als struct abrufbar für konsumierende Services
- [ ] Erste 14 Tage: Profil liefert Defaults (nicht genug Daten)
- [ ] Keine Abhängigkeit von Apple Intelligence

## Technische Vorgaben

### Neue Dateien
| Datei | Beschreibung |
|---|---|
| `BehavioralProfileService.swift` | Berechnet und cached das Verhaltensprofil |
| `BehavioralProfile.swift` | Struct: Profil-Daten (Affinitäten, Kapazität, Patterns) |

### Model
```swift
struct BehavioralProfile {
    let computedDate: Date
    
    // Tageszeit-Affinität: [Kategorie: [Zeitfenster: Anteil]]
    let categoryTimeAffinity: [TaskCategory: [DayPeriod: Double]]
    
    // Tägliche Kapazität
    let avgTasksPerDay: Double
    let avgMinutesPerDay: Double
    
    // Kalender-Korrelation
    let capacityByMeetingLoad: [MeetingLoad: Double] // low/medium/high → avg Tasks
    
    // Schätzgenauigkeit
    let estimationFactor: Double // 1.0 = perfekt, 1.5 = braucht 50% länger
    
    // Verschiebungs-Cluster
    let procrastinationPatterns: [ProcrastinationPattern]
    
    enum DayPeriod: String, CaseIterable { case morning, afternoon, evening }
    enum MeetingLoad: String, CaseIterable { case low, medium, high }
}

struct ProcrastinationPattern {
    let commonCategory: TaskCategory?
    let commonImportance: Int?
    let avgRescheduleCount: Double
    let taskCount: Int
}
```

### Betroffene bestehende Dateien
| Datei | Änderung |
|---|---|

### Constraints
- Berechnung darf max. 500ms dauern (async, nicht auf Main Thread)
- SwiftData-Queries müssen effizient sein: Indices auf `completedDate`, `category`, `isCompleted`
- Profil wird in-memory gecacht, nicht in SwiftData (zu volatil)
- Kein Netzwerk, keine AI, keine externe Dependency
