# Story 0.1: Smart Notification Engine

**Epic:** 0 — Infrastruktur
**Priorität:** Hoch | **Abhängigkeiten:** Keine

## Ziel

Zentraler Service, der alle Notifications der App verwaltet und dabei die harten iOS-Constraints respektiert: max 64 pending Notifications, kein Pre-Delivery-Check, unzuverlässige Background Execution.

## iOS-Constraints (nicht verhandelbar)

- **Max 64 pending local Notifications.** iOS behält nur die 64 zuletzt geplanten. Alles darüber wird still verworfen.
- **Kein Callback vor Zustellung.** Wenn die App nicht läuft, feuern geplante Notifications ungeprüft — auch wenn der Zustand sich geändert hat.
- **Background App Refresh ist unzuverlässig.** BGAppRefreshTask wird vom System ausgeführt wenn es will — keine Garantie für Zeitpunkt oder überhaupt Ausführung.
- **Notifications können entfernt/aktualisiert werden** — aber nur wenn die App aktiv läuft. Per `removePendingNotificationRequests(withIdentifiers:)` oder durch Hinzufügen einer neuen Notification mit derselben Request-ID.

## Kern-Strategie: Reconcile on Event

Statt Notifications weit im Voraus zu planen und zu hoffen: Die App berechnet bei jedem relevanten Event die gesamte Notification-Queue neu.

**Reconciliation-Trigger:**
1. App wird geöffnet (Foreground)
2. Task wird erledigt / erstellt / verschoben
3. Focus Block startet / endet
4. Nutzer ändert Notification-Profil
5. App geht in Background (letzter Sweep)

**Reconciliation-Ablauf:**
1. `removeAllPendingNotificationRequests()`
2. Aktuellen App-Zustand lesen (offene Tasks, geplante Blocks, Kalender, Uhrzeit)
3. Notifications nach Priorität berechnen und einplanen (max 64)

## Akzeptanzkriterien

- [ ] Zentraler `SmartNotificationEngine`-Service ersetzt bestehenden `NotificationService`
- [ ] Reconciliation wird bei jedem der 5 Trigger automatisch aufgerufen
- [ ] Budget-Aufteilung der 64 Slots nach Priorität (siehe unten)
- [ ] Drei Nutzer-Profile konfigurierbar in Settings: Leise, Ausgeglichen, Aktiv
- [ ] Keine Notification enthält vorwurfsvolle/negative Texte (wegen Veralterungs-Risiko)
- [ ] Notification-Texte sind so formuliert, dass sie auch bei verspäteter Zustellung nicht absurd klingen
- [ ] Bestehende interaktive Notification-Actions (NextUp, Postpone, Complete) bleiben erhalten
- [ ] BGAppRefreshTask als Best-Effort-Aufräumer registrieren (aber nicht darauf verlassen)

## Notification-Profile

| Profil | Was wird geplant | Budget ca. |
|---|---|---|
| **Leise** | Nur Focus-Block-Timer (Warning + End-Gong) | 4/64 |
| **Ausgeglichen** | Timer + geplante Task-Erinnerungen (nur heute) + Abend-Review-Nudge + Morning-Nudge | ~25/64 |
| **Aktiv** | Timer + Erinnerungen + Nudges bei Lücken + Micro-Task-Vorschläge | ~50/64 |

## Budget-Prioritäten (bei Reconciliation)

| Prio | Typ | Max Slots | Beschreibung |
|---|---|---|---|
| 1 | Focus Block Timer | 4 | Warning-Gong + End-Gong (aktiver + nächster Block) |
| 2 | Geplante Task-Erinnerungen | 20 | Tasks mit `scheduledDate` heute |
| 3 | Abend-Review / Morning | 2 | Evening Reset Nudge + Morning Coaching |
| 4 | Motivierende Nudges | 10 | Lücken-Vorschläge, Micro-Task-Nudges |
| 5 | Reserve | 28 | Ungenutzt, Puffer für Spitzen |

## Technische Vorgaben

### Neue Dateien
| Datei | Beschreibung |
|---|---|
| `SmartNotificationEngine.swift` | Zentraler Service: Reconciliation-Logik, Budget-Verwaltung, Profil-Auswahl |

### Betroffene bestehende Dateien
| Datei | Änderung |
|---|---|
| `NotificationService.swift` | Refactoring: Wird von SmartNotificationEngine konsumiert oder ersetzt |
| `NotificationActionDelegate.swift` | Bleibt, Actions triggern Reconciliation |
| `AppSettings.swift` | + `notificationProfile: NotificationProfile` (enum: leise/ausgeglichen/aktiv) |
| `SettingsView.swift` | + Notification-Profil-Auswahl |
| `FocusBlockActionService.swift` | Trigger Reconciliation bei Block-Start/Ende |
| `ContentView.swift` | Trigger Reconciliation bei App-Start |

### Identifier-Konvention
```swift
// Alle Notification-IDs folgen diesem Schema für gezieltes Entfernen:
"focusblox.timer.\(blockID)"
"focusblox.task.\(taskID)"
"focusblox.nudge.\(type).\(date)"
"focusblox.review.\(date)"
"focusblox.morning.\(date)"
```

### Constraints
- Reconciliation muss < 100ms dauern (synchron auf Main Thread vermeiden, async berechnen)
- Keine Notification darf Text enthalten, der bei Zustandsänderung peinlich/falsch wird
- BGAppRefreshTask registrieren mit `earliestBeginDate` = nächster voller Stunde, aber App-Logik darf nicht davon abhängen
